# Prompt Maestro para Generar Prototipo Visual Estático — Gamora Bot

## Estado

Documento maestro para solicitar la generación del primer prototipo visual estático de Gamora Bot.

## Propósito

Este documento contiene el prompt rector para producir las pantallas V1 del prototipo visual estático de Gamora Bot, respetando la estrategia, narrativa, flujo, UI Kit y especificación aprobada.

Debe usarse como instrucción base para herramientas de diseño, generación visual, prototipado asistido por IA o trabajo manual de diseño.

No es código.

No es implementación funcional.

No es integración real.

Es una instrucción de producción visual.

---

# Prompt maestro

Actúa como un diseñador senior de producto digital especializado en interfaces SaaS, prototipos visuales, experiencia conversacional y diseño de productos B2B ligeros.

Necesito que construyas el primer prototipo visual estático de **Gamora Bot**, una plataforma que convierte pendientes operativos que nacen en WhatsApp en compromisos estructurados, con responsable, fecha, evidencia, corrección, cierre e historial visible en Web/PWA.

El prototipo debe ser visualmente limpio, profesional, moderno, claro y entendible.

No debe parecer una app de tareas genérica.

No debe parecer un chatbot casual.

No debe parecer un ERP, CRM o sistema corporativo pesado.

No debe parecer una herramienta de vigilancia.

Debe sentirse como una experiencia de operación conversacional estructurada.

---

# Principio rector obligatorio

**WhatsApp opera. Web/PWA estructura, coordina y visualiza. Gamora mantiene el hilo entre ambas.**

Todas las pantallas deben reforzar este principio.

---

# Frase central del producto

**Tus pendientes se atienden en WhatsApp, pero no se pierden en WhatsApp.**

Esta frase debe aparecer como cierre principal del prototipo y puede usarse como mensaje narrativo durante la demo.

---

# Concepto visual rector

## Operación Conversacional Estructurada

La experiencia debe mostrar que una conversación normal puede convertirse en un compromiso operativo con estructura, evidencia, corrección y cierre.

WhatsApp representa la operación rápida y natural.

Web/PWA representa el control visual, la memoria, el historial y la coordinación.

Gamora Bot representa el hilo que conecta ambas interfaces.

---

# Caso demo

## Negocio

Ferretería Luisito.

## Actores

- Luisito: dueño o administrador. Crea, revisa, corrige y cierra compromisos.
    
- Panchito: responsable operativo. Recibe, acepta, ejecuta, evidencia y corrige.
    
- Gamora Bot: asistente operativo. Interpreta, estructura, pregunta, confirma, comunica y registra.
    
- Web/PWA Gamora: capa de control visual. Ordena, consulta, historiza y muestra estados.
    

## Compromiso principal

**Contar los sacos de cemento disponibles en Sucursal Norte y mandar evidencia fotográfica del conteo.**

# Paleta visual sugerida

Usa una paleta profesional basada en:

- Verde operativo profundo: `#147A5A`
    
- Azul petróleo: `#123A4A`
    
- Gris cálido claro: `#F5F7F6`
    
- Blanco suave: `#FFFFFF`
    
- Negro suave: `#16211F`
    
- Gris verdoso: `#6B7A76`
    
- Ámbar operativo para corrección: `#C98218`
    
- Verde de cierre: `#168A5E`
    
- Azul grisáceo para estados informativos: `#4E6E7E`
    
- Borde gris claro: `#DDE5E2`
    

Reglas:

- No uses rojo para correcciones.
    
- No llenes todo de verde.
    
- Usa el color para guiar acciones y estados.
    
- Mantén una estética sobria y confiable.

# Componentes visuales obligatorios

## WhatsApp simulado

Diseña una interfaz tipo móvil, vertical, limpia y familiar, sin copiar de manera literal la marca WhatsApp.

Debe incluir:

- encabezado de chat con “Gamora Bot”,
    
- burbujas de Luisito,
    
- burbujas de Panchito,
    
- tarjetas conversacionales de Gamora,
    
- botones simulados,
    
- adjuntos de evidencia,
    
- badges de estado,
    
- mensajes breves.
    

## Web/PWA

Diseña una interfaz tipo SaaS ligera.

Debe incluir:

- encabezado simple,
    
- tarjetas de compromiso,
    
- estados visibles,
    
- responsable,
    
- fecha compromiso,
    
- evidencia esperada,
    
- panel de evidencia,
    
- historial/timeline,
    
- botón “Volver a WhatsApp”,
    
- botón “Abrir conversación en WhatsApp”.
    

## Conexión entre interfaces

Cuando aparezcan WhatsApp y Web/PWA juntas, muéstralas como dos vistas del mismo flujo.

Usa una línea, flecha suave o microcopy como:

**Una acción. Dos vistas. Un solo compromiso.**

o

**Gamora mantiene el hilo.**

# Estados visuales

Usa estos estados:

- Borrador.
    
- Nuevo.
    
- Aceptado.
    
- Evidencia enviada.
    
- Necesita corrección.
    
- Corregido.
    
- Pendiente de cierre.
    
- Cerrado.
    

Reglas:

- “Necesita corrección” debe usar ámbar suave.
    
- “Cerrado” debe usar verde sólido.
    
- Los estados deben ser visibles, pero no dominar la pantalla.
    
- Evitar lenguaje como “rechazado”, “error”, “incumplimiento” o “no válido”.

# Reglas visuales obligatorias

1. Mantener claridad antes que decoración.
    
2. Mostrar WhatsApp como interfaz operativa.
    
3. Mostrar Web/PWA como capa de control.
    
4. No hacer Web/PWA genérica.
    
5. No saturar pantallas móviles.
    
6. No agregar funcionalidades fuera del flujo aprobado.
    
7. No usar rojo en correcciones.
    
8. No mostrar lectura de chats personales ni grupos.
    
9. No presentar a Gamora como IA autónoma.
    
10. No convertirlo en task manager.
    
11. Mantener estados visibles.
    
12. Mostrar evidencia con peso visual.
    
13. Mostrar cierre humano.
    
14. Mostrar continuidad entre interfaces.
    

---

# Resultado esperado

El resultado debe ser una secuencia visual profesional y estática que permita presentar Gamora Bot en menos de 5 minutos.

La demo debe dejar claro que:

- WhatsApp es la interfaz natural de operación.
    
- Gamora convierte mensajes en compromisos.
    
- La Web/PWA estructura, coordina e historiza.
    
- La evidencia queda asociada.
    
- La corrección conserva el hilo.
    
- El cierre requiere aprobación humana.
    
- La experiencia se siente como una sola, aunque use dos interfaces.
    

---

# Formato de entrega esperado

Entregar como:

- serie de pantallas estáticas,
    
- o slides,
    
- o prototipo visual no funcional,
    
- o storyboard visual de alta fidelidad media.
    

Cada pantalla debe mantener consistencia visual y narrativa.

---

# Criterios de aprobación

El prototipo visual generado será aceptable si:

- se entiende sin explicación larga,
    
- se ve profesional,
    
- sigue las 15 pantallas,
    
- respeta la paleta y tono visual,
    
- no agrega funciones no aprobadas,
    
- muestra claramente WhatsApp + Web/PWA,
    
- hace que Web/PWA se sienta necesaria,
    
- no parece task manager,
    
- no parece vigilancia,
    
- no promete capacidades inviables,
    
- genera deseo de probarlo.
    

---

# Instrucción final

Construye el prototipo visual como si fuera una primera demo seria para validar con dueños de PyME, coordinadores operativos o responsables que ya usan WhatsApp para dar seguimiento a pendientes.

El diseño debe sentirse real, simple y profesional.

No busques impresionar con efectos visuales.

Busca que el usuario piense:

**“Esto sí me resolvería el problema de los pendientes que se me pierden en WhatsApp.”**
