# Preparación para Prototipo Visual — Gamora Bot

## Estado

Documento rector previo al diseño visual del prototipo simulado de Gamora Bot.

## Propósito

Este documento consolida las piezas ya aprobadas y define las reglas que deberá seguir el prototipo visual de Gamora Bot.

Su objetivo es evitar pasar a diseño visual sin una base clara de narrativa, flujo, experiencia, validación y viabilidad.

Este documento no es diseño visual final.

No define colores definitivos.

No define tipografía definitiva.

No define identidad visual completa.

No define prototipo funcional.

Su función es preparar el terreno para diseñar visualmente sin perder claridad estratégica.

---

# Principio rector aprobado

**WhatsApp opera. Web/PWA estructura, coordina y visualiza. Gamora mantiene el hilo entre ambas.**

Este principio debe guiar todo el prototipo visual.

---

# Frase central de producto

**Tus pendientes se atienden en WhatsApp, pero no se pierden en WhatsApp.**

Esta frase debe mantenerse como eje narrativo del prototipo.

---

# Estado actual de la etapa

Gamora Bot se encuentra en etapa de prototipo simulado, estático y sin código.

No se ejecutará MVP 3.

No se involucrará Codex.

No se desarrollará backend, integración real con WhatsApp, IA real ni Web/PWA funcional.

La prioridad es validar claridad, deseo de uso, fricción percibida y valor comercial.

# Decisión estratégica vigente

La siguiente etapa no debe ser desarrollo técnico.

La siguiente etapa tampoco debe ser diseño visual disruptivo final sin control.

La siguiente etapa debe ser la preparación y construcción de un prototipo visual estático, ligero y entendible, basado en los wireframes y storyboard aprobados.

---

# Objetivo del prototipo visual

El prototipo visual debe permitir que un usuario entienda en pocos minutos:

1. Qué problema resuelve Gamora Bot.
    
2. Por qué WhatsApp normal no basta para cerrar compromisos.
    
3. Cómo Gamora estructura una instrucción natural.
    
4. Cómo Panchito opera con baja fricción.
    
5. Cómo Luisito conserva control.
    
6. Cómo la evidencia queda asociada.
    
7. Cómo la corrección no rompe el flujo.
    
8. Cómo el cierre queda confirmado.
    
9. Por qué Web/PWA no sobra.
    
10. Cómo WhatsApp y Web/PWA se sienten como una sola experiencia.
    

---

# Lo que el prototipo visual debe demostrar

El prototipo debe demostrar:

- creación de compromiso desde WhatsApp,
    
- interpretación y estructuración por Gamora,
    
- confirmación humana antes de crear,
    
- asignación a Panchito,
    
- aceptación desde WhatsApp,
    
- evidencia enviada desde WhatsApp,
    
- revisión desde WhatsApp o Web/PWA,
    
- corrección estructurada,
    
- evidencia corregida,
    
- aprobación y cierre humano,
    
- historial final en Web/PWA,
    
- conexión bidireccional entre ambas interfaces.
    

---

# Lo que el prototipo visual no debe demostrar todavía

El prototipo no debe intentar demostrar:

- login,
    
- registro,
    
- administración avanzada,
    
- multiempresa,
    
- configuración compleja,
    
- reportes ejecutivos,
    
- dashboard lleno de métricas,
    
- inteligencia artificial autónoma,
    
- lectura de chats personales,
    
- grupos de WhatsApp,
    
- integración real con Meta,
    
- base de datos real,
    
- backend,
    
- APIs,
    
- pricing,
    
- cobranza,
    
- marketplace,
    
- analítica avanzada.
    

---

# Regla de diseño visual

El diseño visual debe venir después de la claridad.

La estética debe reforzar el flujo, no competir con él.

La prioridad será:

1. Comprensión.
    
2. Fluidez.
    
3. Confianza.
    
4. Baja fricción.
    
5. Sensación de una sola experiencia.
    
6. Belleza visual.
    
7. Disrupción.
    

---

# Regla de doble interfaz

El prototipo visual debe mostrar que WhatsApp y Web/PWA no son dos productos separados.

Deben sentirse como dos caras de un mismo flujo.

## WhatsApp

Debe sentirse:

- natural,
    
- rápido,
    
- cotidiano,
    
- conversacional,
    
- operativo.
    

## Web/PWA

Debe sentirse:

- ordenada,
    
- visual,
    
- confiable,
    
- estructurada,
    
- útil,
    
- no obligatoria.
    

---

# Momentos donde debe aparecer WhatsApp

WhatsApp debe aparecer en:

1. El problema inicial.
    
2. La instrucción natural de Luisito.
    
3. La estructuración de Gamora.
    
4. La confirmación de creación.
    
5. La recepción de Panchito.
    
6. La aceptación de Panchito.
    
7. El envío de evidencia.
    
8. La revisión de Luisito.
    
9. La corrección.
    
10. El reenvío de evidencia.
    
11. La aprobación y cierre.
    
12. La confirmación final.
    

---

# Momentos donde debe aparecer Web/PWA

Web/PWA debe aparecer en tres momentos clave:

## 1. Después de crear el compromiso

Objetivo:

Mostrar que lo que nació en WhatsApp queda estructurado.

## 2. Durante la revisión de evidencia

Objetivo:

Mostrar vista ampliada, contexto y decisión ordenada.

## 3. Al final con historial

Objetivo:

Mostrar memoria operativa, trazabilidad y control.

---

# Regla de transición

Cada transición debe conservar contexto.

## WhatsApp hacia Web/PWA

No debe abrir una pantalla genérica.

Debe abrir el compromiso específico.

## Web/PWA hacia WhatsApp

No debe mandar a una conversación vacía.

Debe regresar al hilo o acción relacionada.

---

# Mensaje visual que debe sentirse

El usuario debe sentir:

“Estoy trabajando en WhatsApp, pero Gamora está ordenando todo por detrás.”

Y también:

“Cuando necesito ver más, la plataforma me muestra todo claro.”

---

# Estilo visual recomendado para primera versión

La primera versión visual no debe ser excesivamente decorativa.

Debe ser:

- limpia,
    
- clara,
    
- moderna,
    
- ligera,
    
- confiable,
    
- visualmente simple,
    
- enfocada en flujo,
    
- con pocos elementos por pantalla.
    

---

# Personalidad visual sugerida

Gamora Bot debe sentirse como:

- asistente operativo,
    
- confiable,
    
- rápido,
    
- humano,
    
- moderno,
    
- no corporativo pesado,
    
- no infantil,
    
- no invasivo,
    
- no futurista exagerado.
    

---

# Riesgos visuales a evitar

## Riesgo 1 — Hacerlo demasiado bonito antes de hacerlo claro

El diseño no debe ocultar problemas de flujo.

## Riesgo 2 — Hacer que parezca un chatbot genérico

Gamora no es un bot conversacional cualquiera.

Es un asistente operativo de compromisos.

## Riesgo 3 — Hacer que parezca task manager

Evitar que el prototipo se vea como Trello, Asana, Monday o lista de tareas tradicional.

## Riesgo 4 — Hacer que Web/PWA parezca secundaria sin valor

La Web/PWA debe aparecer poco, pero con mucho poder.

## Riesgo 5 — Hacer que WhatsApp parezca invadido

El flujo debe verse natural, no saturado de mensajes.

## Riesgo 6 — Prometer cosas inviables

No mostrar lectura de chats personales, grupos ni automatizaciones no confirmadas.

# Pantallas/momentos visuales mínimos

Para una primera demo visual, se recomiendan estas 10 piezas:

1. Antes: WhatsApp desordenado.
    
2. Luisito dicta instrucción a Gamora.
    
3. Gamora estructura y pide confirmación.
    
4. Web/PWA muestra compromiso creado.
    
5. Panchito recibe y acepta.
    
6. Panchito manda evidencia.
    
7. Luisito revisa y puede abrir Web/PWA.
    
8. Luisito pide corrección y Panchito corrige.
    
9. Luisito aprueba y cierra.
    
10. Web/PWA muestra historial final.
    

---

# Versión visual compacta

Si se necesita una versión aún más breve, usar 6 escenas:

1. Problema en WhatsApp.
    
2. Instrucción natural a Gamora.
    
3. Compromiso estructurado.
    
4. Evidencia y corrección.
    
5. Cierre.
    
6. Historial en Web/PWA.
    

---

# Versión visual completa

Si se quiere mostrar el flujo entero, usar las 14 escenas del storyboard aprobado.

La versión completa debe reservarse para una demo más pausada.

---

# Jerarquía narrativa visual

El prototipo debe contar esta historia:

## Antes

WhatsApp tiene conversación, pero no control.

## Cambio

Gamora convierte lenguaje natural en compromiso.

## Operación

Panchito acepta y evidencia desde WhatsApp.

## Control

Luisito revisa, corrige y cierra.

## Memoria

Web/PWA conserva todo ordenado.

---

# Criterios para evaluar el prototipo visual

El prototipo visual será correcto si:

- se entiende sin explicación larga,
    
- muestra baja fricción,
    
- no parece app de tareas,
    
- muestra valor claro de WhatsApp,
    
- muestra valor claro de Web/PWA,
    
- permite entender evidencia, corrección y cierre,
    
- transmite confianza,
    
- no parece invasivo,
    
- no promete lectura de chats,
    
- genera ganas de probarlo.

El proyecto está listo para preparar el primer prototipo visual estático.

Sin embargo, el diseño visual debe iniciar con una versión clara y funcional, no con una versión excesivamente disruptiva.

La disrupción visual deberá agregarse después de validar que la historia se entiende.

# Cierre

Gamora Bot ya tiene una base estratégica, narrativa y funcional suficientemente clara para avanzar hacia prototipo visual.

La regla será no embellecer la confusión.

Solo se diseñará visualmente lo que ya fue validado como flujo claro.

El objetivo del prototipo visual será demostrar, con la menor fricción posible, que Gamora Bot convierte pendientes operativos de WhatsApp en compromisos estructurados, evidenciables, corregibles y cerrables, conectados con una Web/PWA que ordena y conserva todo el control.
